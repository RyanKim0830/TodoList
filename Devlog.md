# 개발 일지

## 디렉토리 구조 결정

```
app/
├── main.py          # FastAPI 앱 시작점
├── models.py         # DB 모델 (User, Todo) -> DB에 저장되는 형태
├── schemas.py        # Pydantic 스키마 (요청/응답 형태) - API 로 주고받는 형태
├── database.py       # DB 연결 설정
├── auth.py           # JWT 관련 로직
└── routers/
    ├── auth.py        # 회원가입/로그인 라우터
    └── todos.py       # Todo CRUD 라우터
```

위와 같이 설계 한 이유 -> 표준 관례라고 함.
```main.py       → 앱을 "시작"만 시킴. 라우터들을 모아서 등록하는 진입점
models.py     → DB 테이블 구조 정의 (SQLAlchemy가 이걸 보고 실제 테이블 만듦)
schemas.py    → API가 주고받는 데이터 "모양" 정의 (Pydantic)
database.py   → DB 연결 설정 (커넥션, 세션 관리)
auth.py       → JWT 생성/검증 로직 (재사용될 함수들이라 따로 뺌)
routers/      → 실제 엔드포인트들 (도메인별로 auth, todos 나눔)
```

### 개발 단계
> 뭐부터 건드려야 할지 모르곘어서 클로드한테 순서좀 알려달라 함
```
클로드의 답변
1. 모델 설계 (User, Todo가 어떤 필드 가질지) (DB 테이블 구조 정의)
2. DB 연결 설정
3. 스키마 정의 (API로 주고받을 데이터 모양)
4. 회원가입/로그인 라우터
5. Todo CRUD 라우터
6. 인증 연결 (내 Todo만 보이게)
7. Swagger UI로 직접 테스트
8. (선택) 필터링/페이지네이션 추가
```
``` 
이유
1→2: 모델이 있어야 DB에 뭘 저장할지 알 수 있음
2→3: DB 저장 형태 정해지면, API로 주고받을 형태(스키마)도 정할 수 있음
3→4: 회원가입부터 하는 이유 = Todo보다 간단하고, 
     이게 있어야 5번에서 "누구의 Todo인지" 구분 가능
4→5: Todo CRUD, 이때는 아직 인증 없이 "모두의 Todo"로 단순하게 먼저 만듦
5→6: 다 만든 다음에 "로그인한 사람 것만 보이게" 인증을 얹음
     (처음부터 인증까지 같이 하면 복잡도가 한 번에 몰려서 헷갈림)
6→7: 다 만들고 나서 실제로 되는지 확인
```
> "한 번에 다 짜지 말고, 기능 하나를 완전히 되게 만든 다음에 그 위에 다음 기능을 얹어라"
## 모델 설계 (User, Todo의 필드 설계)
- Todo의 필드 : 이름(str), 완료 여부 (bool), 마감기한(datetime), id(고유번호)
- User의 필드 : 이름(str), ID(str), pw_hash(str): 누구도 pw 원본 저장하는 정신나간짓은 안한다.
이때 'SQLAlchemy라는 DB ORM 라이브러리를 사용하였다, DB를 python으로 조작 하기 위함.'


## 개발 중 모르는 것들이 존나 많았음

- 필드 : 데이터 어떤 정보조각들 이루어지는지 -> "사람"이라는 물건의 필드 : "이름", "나이", "키" 등
